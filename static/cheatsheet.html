🗄️ SQL — core

  • Logical order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT.
  • WHERE filters rows before grouping; HAVING filters groups after aggregation.
  • NULL = unknown: test with IS NULL, never =. COALESCE(x,fallback) for defaults.
  • COUNT(*) counts rows; COUNT(col) skips NULLs; COUNT(DISTINCT col) uniques.

🔗 SQL — joins

  • INNER = matches only. LEFT = all left + NULLs for non-matches.
  • Anti-join (find missing): LEFT JOIN … WHERE right.key IS NULL.
  • ON vs WHERE on a LEFT join: filtering the right table in WHERE silently turns it into an INNER join — filter it in ON.
  • Joining a non-unique key multiplies rows — know each side's grain before you SUM.

🪟 Window functions

func() OVER (PARTITION BY ..
            ORDER BY ..)
  • ROW_NUMBER (1,2,3) · RANK (gaps) · DENSE_RANK (no gaps).
  • LAG/LEAD = prev/next row → period-over-period change.
  • SUM() OVER(ORDER BY) = running total.
  • Top-N per group: ROW_NUMBER in a subquery, then WHERE rn<=N (can't filter the window in the same SELECT).
  • Dedup: ROW_NUMBER() … ORDER BY updated_at DESC, keep rn=1.

⚡ SQL — performance

  • EXPLAIN first; watch for full table scan vs index scan.
  • Index the columns you filter/join/sort on.
  • Don't wrap an indexed column in a function in WHERE (kills the index) — use a range.
  • Select only needed columns; filter early; partition big tables.
  • Cost ∝ data scanned — scanning 100× less is ~100× faster and cheaper.

🐍 Python for DE

  • Comprehensions: [f(x) for x in xs if cond].
  • Generators (yield) = lazy, constant memory → process data > RAM.
  • try/except specific errors; finally always runs (cleanup).
  • pandas: boolean mask = WHERE, groupby().agg() = GROUP BY, merge() = JOIN. Vectorise; avoid iterrows.
  • Store timestamps in UTC; convert to local only for display.

📐 Data modeling

  • Fact = events/measures + keys (tall, skinny). Dimension = context (short, wide).
  • Grain = what one fact row means — decide it first, in one sentence.
  • Star (denormalized dims, fewer joins) vs Snowflake (normalized).
  • Surrogate keys (generated ints) over business keys; enables history.
  • SCD2 = new dim row + validity dates to keep history.
  • Measures: additive / semi-additive (not over time) / non-additive (ratios).

🧱 OLTP vs OLAP · formats

  • OLTP = many small txns, row-store. OLAP = big scans, column-store.
  • Columnar (Parquet) reads only needed columns + compresses well → 10–100× for analytics.
  • Normalize for write-heavy OLTP; denormalize for read-heavy analytics.
  • CSV (simple, untyped) · JSON (nested, APIs) · Parquet (columnar, analytics) · Avro (row, streaming).

⚙️ Spark

  • Transformations (filter/select/join) are lazy; actions (count/write/collect) trigger the DAG.
  • Laziness → Catalyst optimises the whole plan (pushdown, column pruning).
  • Partitions = parallel chunks (~128–256 MB, ~1/core).
  • Shuffle (groupBy/join/distinct) moves data across network — the main cost. Filter early.
  • Skew: one hot key → 1 slow task. Fix: salting, AQE, broadcast.
  • Broadcast small table in a join to skip the shuffle. cache() reused DataFrames.

🔧 Pipelines

  • ETL (transform before load) vs ELT (load raw, transform in-warehouse — modern default).
  • Idempotent = re-run = same result. Use upsert/MERGE or delete-then-insert per partition.
  • Incremental via high-water mark; CDC reads the DB change log (captures deletes).
  • Backfill = re-run past dates; needs date-parameterised + idempotent runs.
  • Orchestration (Airflow): DAG of tasks with dependencies, schedules, retries.

🌊 Streaming

  • Event time (when it happened) vs processing time (when handled) — window by event time.
  • Windows: tumbling (fixed), sliding (overlap), session (activity gaps).
  • Watermark = max event time − grace; passes window end → finalise. Later = late data.
  • Guarantees: at-most-once (lossy) · at-least-once (dups) · exactly-once (once).
  • Exactly-once = checkpoints + idempotent/transactional sink. Cheap path: at-least-once + upsert by id.

🏞️ Lakehouse

  • Lake (cheap files, no safety) + warehouse (reliable, costly) → lakehouse.
  • Table formats (Delta/Iceberg/Hudi) add a transaction log over files →
  • ACID writes · row-level UPDATE/DELETE/MERGE (GDPR!) · time travel · schema evolution.
  • Compaction fixes the small-files problem; vacuum reclaims storage.
  • Medallion: bronze (raw) → silver (clean) → gold (business-ready).

☁️ Cloud

  • Compute ladder: VM (most control) → container/K8s → serverless (scale-to-zero).
  • Object storage (S3/GCS/ADLS) = cheap, scalable backbone.
  • Cost: scan less (partition + columnar + narrow SELECT), tier old data, co-locate compute (avoid egress), use spot for batch.
  • IAM: least privilege via roles; service accounts for pipelines; encrypt at rest + in transit.

🧩 NoSQL & CAP

  • Key-value (fetch by key) · document (flexible JSON) · wide-column (huge writes) · graph (relationships).
  • CAP: during a partition, choose C or A (P is unavoidable). CP = correct; AP = available + eventual consistency.
  • Sharding splits data by key (handles size); replication copies it (reliability + read scale).
  • Pick the store by access pattern, not hype.

🛠️ DataOps & quality

  • Observability pillars: freshness, volume, schema, distribution, lineage.
  • Quality checks: not-null, uniqueness, range/valid-set, referential — block publish on fail.
  • Beware silent failures ("succeeded" but wrong/empty data).
  • CI/CD runs tests per change; IaC (Terraform) = version-controlled infra; SLA = promise to consumers.

🛡️ Governance

  • PII = identifies a person (directly or via quasi-identifiers).
  • GDPR rights: access & erasure ("be forgotten") → needs row-level delete + lineage.
  • RBAC + least privilege; restrict at table/column/row level.
  • Masking (****1234), tokenization, pseudonymization for analytics.
  • Data residency = keep data in-region.

🎯 System design method

  • 1 Clarify (freshness, scale, users, history).
  • 2 Estimate: vol = events×bytes; tput = events÷86,400 ×peak(2–5); storage = daily×retention×overhead.
  • 3 Flow: source → ingest → store → transform → serve.
  • 4 Choose components with reasons.
  • 5 Hard parts: idempotency, late/dup data, quality, schema change, monitoring.
  • 6 Trade-offs: simplest design that meets the requirement (batch unless real-time is truly needed).

⚡ Rapid-fire Q&A

Find customers with no orders?

LEFT JOIN orders … WHERE order_id IS NULL.

2nd-highest salary?

DENSE_RANK() OVER(ORDER BY salary DESC), pick rank=2.

WHERE vs HAVING?

WHERE = rows pre-group; HAVING = groups post-aggregate.

Why is shuffle slow?

Moves data across network/disk by key.

Make a pipeline idempotent?

Upsert/MERGE or delete-then-insert per partition.

Exactly-once?

Checkpoints + idempotent/transactional sink.

Delete one user from a lake?

Use a table format (Delta/Iceberg) with row-level DELETE + lineage.

⚡ More Q&A

ETL vs ELT?

Transform before vs after loading; ELT suits cloud warehouses.

Star vs snowflake?

Denormalized dims (fewer joins) vs normalized.

CAP for a shopping cart?

AP — stay available, reconcile; brief staleness is fine.

Fix Spark data skew?

Salting, AQE skew handling, or broadcast the small side.

Parquet vs CSV?

Columnar → reads only needed columns, compresses, far faster.

Slowly Changing Dimension type 2?

New row per change with validity dates + current flag.

Catch bad data before it ships?

Quality assertions + freshness/volume checks that block publish.
