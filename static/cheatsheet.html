Foundations & database systems

🧱 Data engineering foundations

  • DE builds the systems that move, store & shape data; analysts query it, scientists model it.
  • OLTP = many small txns (row-store, normalized). OLAP = big scans (column-store, denormalized).
  • Batch (bounded, scheduled) vs streaming (unbounded, continuous). Pick batch unless latency truly demands stream.
  • Latency vs throughput vs cost is the constant trade-off — decide from the requirement.

📦 Bytes, files & formats

  • Row (CSV, JSON, Avro) = fast writes/whole-row reads. Columnar (Parquet, ORC) = analytics: read only needed cols, compress 10–100×.
  • Avro row + schema (streaming/Kafka); Parquet columnar (lake); ORC columnar (Hive).
  • Compression: Snappy (fast, splittable) vs Gzip/Zstd (smaller). Splittable matters for parallelism.
  • Store text as UTF-8; timestamps in UTC; watch encoding & timezone bugs.

🗃️ DBMS internals

  • ACID: Atomic (all/nothing), Consistent (rules hold), Isolated (concurrent = serial-like), Durable (survives crash).
  • Isolation levels: read-committed → repeatable-read → serializable; higher = fewer anomalies, more locking. Anomalies: dirty/non-repeatable/phantom read.
  • Index = B-tree → O(log n) lookups/ranges; costs write speed + storage.
  • WAL (write-ahead log) gives durability + crash recovery; MVCC = readers don't block writers.

🧮 Relational model & normalization

  • Keys: primary (unique, not null), foreign (references another table), candidate/surrogate.
  • 1NF atomic cells · 2NF no partial-key dependency · 3NF no transitive dependency.
  • Normalize to kill update anomalies (OLTP); denormalize for read speed (OLAP).
  • Referential integrity via FKs; ON DELETE CASCADE/RESTRICT controls dependents.
SQL

🗄️ SQL — core

  • Logical order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT.
  • WHERE filters rows before grouping; HAVING filters groups after aggregation.
  • NULL = unknown: test with IS NULL, never =. COALESCE(x,fallback) for defaults.
  • COUNT(*) counts rows; COUNT(col) skips NULLs; COUNT(DISTINCT col) uniques.

🔗 SQL — joins

  • INNER = matches only. LEFT = all left + NULLs for non-matches. FULL = both sides.
  • Anti-join (find missing): LEFT JOIN … WHERE right.key IS NULL.
  • ON vs WHERE on a LEFT join: filtering the right table in WHERE silently makes it INNER — filter it in ON.
  • Joining a non-unique key multiplies rows — know each side's grain before you SUM.

🪟 Window functions

func() OVER (PARTITION BY ..
            ORDER BY ..)
  • ROW_NUMBER (1,2,3) · RANK (gaps) · DENSE_RANK (no gaps).
  • LAG/LEAD = prev/next row → period-over-period change.
  • SUM() OVER(ORDER BY) = running total; frames: ROWS BETWEEN … .
  • Top-N per group: ROW_NUMBER in a subquery, then WHERE rn<=N.
  • Dedup: ROW_NUMBER() … ORDER BY updated_at DESC, keep rn=1.

🔤 SQL — functions & types

  • CASE WHEN for conditional logic; NULLIF, COALESCE for null handling.
  • Dates: DATE_TRUNC, DATEDIFF, EXTRACT; store UTC, bucket by day/month.
  • Strings: TRIM, SUBSTRING, REPLACE, LIKE '%x', SPLIT_PART.
  • CAST/::type; beware implicit casts. GROUP BY GROUPING SETS/ROLLUP/CUBE for subtotals.
  • CTEs (WITH) for readability; recursive CTEs for hierarchies.

✏️ SQL — DDL/DML & MERGE

  • DDL: CREATE/ALTER/DROP; DML: INSERT/UPDATE/DELETE. Constraints: PK, FK, UNIQUE, CHECK, NOT NULL.
  • MERGE / upsert = insert-or-update by key — the idempotency workhorse.
  • Wrap multi-step writes in a transaction; commit atomically or roll back.
  • Views (saved query) vs materialized views (stored, refreshed).

⚡ SQL — performance

  • EXPLAIN first; watch full scan vs index scan and join order.
  • Index the columns you filter/join/sort on; don't wrap them in functions in WHERE (kills the index).
  • Select only needed columns; filter early; partition big tables (prune by date).
  • Cost ∝ data scanned — scanning 100× less is ~100× faster and cheaper.
Data modeling & warehousing

📐 Dimensional modeling

  • Fact = events/measures + keys (tall, skinny). Dimension = context (short, wide).
  • Grain = what one fact row means — decide it first, in one sentence.
  • Star (denormalized dims, fewer joins) vs Snowflake (normalized dims).
  • Surrogate keys (generated ints) over business keys; enables history & decoupling.
  • Measures: additive / semi-additive (not over time) / non-additive (ratios).

🏛️ Modeling approaches & SCD

  • Kimball (bottom-up star marts) vs Inmon (top-down 3NF EDW) vs Data Vault (hubs/links/sats, audit-heavy).
  • Data Vault 2.0: Hub (business key), Link (relationship), Satellite (descriptive + history) — highly parallel/auditable.
  • SCD1 overwrite · SCD2 new row + validity dates + current flag (keep history) · SCD3 prior-value column.
  • OBT / wide tables (one big table) trade storage for zero-join speed in columnar warehouses.
Python for data engineering

🐍 Python for DE

  • Comprehensions: [f(x) for x in xs if cond].
  • Generators (yield) = lazy, constant memory → process data > RAM.
  • Context managers (with) for cleanup; try/except specific errors, finally always runs.
  • Type hints + mypy; dataclasses/pydantic for typed records & validation.
  • Store timestamps in UTC; convert to local only for display.

🧮 Python data stack

  • pandas: boolean mask = WHERE, groupby().agg() = GROUP BY, merge() = JOIN. Vectorize; avoid iterrows.
  • NumPy = vectorized arrays (C speed). Polars = fast, multi-threaded, lazy DataFrame.
  • Arrow = in-memory columnar standard (zero-copy interchange); Parquet = on-disk columnar.
  • DuckDB = in-process OLAP SQL on files; Dask = parallel/out-of-core pandas.

🔌 Ingestion & validation

  • requests/APIs: paginate, rate-limit, retry with backoff, handle 429/5xx.
  • SQLAlchemy for DB access; parameterize queries (no string-format → SQL injection).
  • pydantic/Great Expectations to validate schema & values at the boundary.
  • Structured logging + retries + idempotent writes = reliable ingestion.

🐚 Unix & shell

  • Pipes: cat f | grep x | sort | uniq -c | sort -rn.
  • grep (search), awk (columns), sed (substitute), cut/sort/uniq/wc.
  • cron * * * * * = min hr dom mon dow; exit code 0=success.
  • xargs for batching; set -euo pipefail for safe scripts; chmod/permissions.
NoSQL, big data & Spark

🧩 NoSQL & CAP

  • Key-value (fetch by key) · document (JSON) · wide-column (huge writes, Cassandra) · graph (relationships).
  • CAP: during a partition, choose C or A (P unavoidable). CP = correct; AP = available + eventual.
  • PACELC: else (no partition) trade Latency vs Consistency.
  • Sharding splits by key (size); replication copies (reliability + read scale). Pick store by access pattern.

⚙️ Spark — model & internals

  • Transformations (filter/select/join) are lazy; actions (count/write/collect) trigger the DAG.
  • Laziness → Catalyst optimizes the whole plan (predicate pushdown, column pruning).
  • Partitions = parallel chunks (~128–256 MB, ~1/core). Narrow deps (map/filter) vs wide (shuffle).
  • DataFrame API > RDD (Catalyst + Tungsten); same speed across Python/Scala.

🔀 Spark — shuffle, skew, joins

  • Shuffle (groupBy/join/distinct/repartition) moves data across network — the main cost. Filter early.
  • Skew: one hot key → 1 slow task. Fix: salting, AQE skew join, broadcast.
  • Broadcast small table (< ~10–100 MB) to skip the shuffle; SortMerge for two big sides.
  • cache() reused DataFrames; tune partition count; avoid tiny files & collect() on big data.

🐘 Big data ecosystem

  • MapReduce = map → shuffle → reduce (disk-heavy); Spark = in-memory DAG, ~10–100× faster.
  • HDFS = distributed block storage (replicated); YARN = cluster resource manager.
  • Hive = SQL over files + metastore (schema-on-read); MSCK REPAIR/ADD PARTITION registers new files.
  • Cloud object storage (S3/GCS) largely replaced HDFS; compute is now elastic & separate.
Pipelines, orchestration & dbt

🔧 Pipelines

  • ETL (transform before load) vs ELT (load raw, transform in-warehouse — modern default).
  • Idempotent = re-run = same result. Use upsert/MERGE or delete-then-insert per partition.
  • Incremental via high-water mark; CDC reads the DB change log (captures deletes too).
  • Backfill = re-run past dates; needs date-parameterized + idempotent runs.

🗺️ Orchestration (Airflow)

  • DAG = tasks + dependencies + schedule; operators do work, sensors wait for conditions.
  • Idempotent, atomic tasks keyed by execution_date → safe retries & backfills.
  • Retries + alerts + SLAs; avoid heavy work in the scheduler; pass data via storage, not XCom.
  • Modern: Dagster/Prefect (asset-/flow-centric, typed, local dev); Airflow = the incumbent.

🧱 dbt — analytics engineering

  • Models = SELECTs; ref()/source() build the DAG & lineage automatically.
  • Materializations: view · table · incremental (append/merge new rows) · ephemeral.
  • Tests (unique, not_null, relationships, accepted_values) + snapshots = SCD2 history.
  • Macros (Jinja) for DRY SQL; docs + exposures; runs in-warehouse (ELT).
Streaming & real-time

🌊 Streaming fundamentals

  • Event time (when it happened) vs processing time (when handled) — window by event time.
  • Windows: tumbling (fixed), sliding (overlap), session (activity gaps).
  • Watermark = max event time − grace; passes window end → finalize. Later = late data.
  • Guarantees: at-most-once (lossy) · at-least-once (dups) · exactly-once. Cheap path: at-least-once + upsert by id.

📨 Kafka & platforms

  • Topic → partitions (ordering per partition, by key); offset = position; consumers commit offsets.
  • Consumer group = parallelism (1 partition → 1 consumer); replication factor for durability.
  • Exactly-once = idempotent producer + transactions, or idempotent sink + checkpoints.
  • Processors: Flink (true streaming, rich state), Spark Structured Streaming (micro-batch); managed: Kinesis, Pub/Sub.
Lakehouse & cloud

🏞️ Lakehouse & table formats

  • Lake (cheap files, no safety) + warehouse (reliable, costly) → lakehouse.
  • Table formats (Delta / Iceberg / Hudi) add a transaction log over Parquet → ACID, row-level UPDATE/DELETE/MERGE (GDPR!), time travel, schema evolution.
  • Compaction / OPTIMIZE fixes the small-files problem; vacuum reclaims old versions.
  • Medallion: bronze (raw) → silver (clean/conformed) → gold (business-ready).

☁️ Cloud foundations

  • Compute ladder: VM (most control) → container/K8s → serverless (scale-to-zero).
  • Object storage (S3/GCS/ADLS) = cheap, durable, scalable backbone; separate storage from compute.
  • Cost: scan less (partition + columnar + narrow SELECT), tier cold data, avoid cross-region egress, use spot/preemptible for batch.
  • IAM: least privilege via roles; service accounts for pipelines; encrypt at rest (KMS) + in transit.
Platforms — Snowflake · Databricks · AWS · GCP

❄️ Snowflake — architecture

  • 3 layers: cloud services (optimizer, metadata, security) · compute (virtual warehouses) · storage (columnar micro-partitions) — storage & compute scale independently.
  • Virtual warehouse = MPP compute cluster; per-second billing, auto-suspend/resume; size up (bigger) vs multi-cluster (more concurrency).
  • Micro-partitions (~50–500 MB, immutable) carry min/max metadata → automatic pruning; add a clustering key on huge tables.
  • Caching: result cache (24 h, free) → warehouse SSD cache → remote storage.

❄️ Snowflake — features

  • Snowpipe = continuous, serverless auto-ingest from cloud storage (event-driven).
  • Streams (table CDC) + Tasks (scheduled SQL) = pipelines; Dynamic Tables = declarative incremental transforms.
  • Time Travel (query/restore up to 90 d) + Fail-safe (7 d); zero-copy clone = instant metadata-only copy.
  • RBAC (roles, grants, future grants); secure data sharing / Marketplace (no copy); VARIANT + FLATTEN for JSON.

🧱 Databricks & Delta Lake

  • Delta Lake = Parquet + _delta_log → ACID, MERGE, time travel, schema evolution; OPTIMIZE + ZORDER, VACUUM.
  • Structured Streaming + Auto Loader (cloudFiles) = incremental ingest, exactly-once to Delta via checkpoints.
  • Unity Catalog = central governance (catalog.schema.table, lineage, ACLs); DLT = declarative pipelines with expectations.
  • Photon = vectorized engine; clusters (all-purpose vs job); medallion bronze/silver/gold.

🟧 AWS for data engineering

  • S3 = storage backbone (prefixes, storage classes, lifecycle). Glue = Data Catalog + serverless Spark ETL + crawlers + bookmarks (incremental).
  • Athena = serverless Trino SQL on S3 (pay per TB scanned → partition + Parquet); CTAS to transform.
  • Redshift = MPP warehouse (dist key + sort key); Spectrum queries S3 directly.
  • Kinesis/Firehose (streaming/delivery), Lake Formation (fine-grained lake perms), Step Functions/EMR (orchestration/Spark).

🟦 GCP for data engineering

  • BigQuery = serverless MPP warehouse; bills per bytes scanned (or slots); partition (date) + cluster (sort) to prune; storage/compute separate.
  • Dataflow = managed Apache Beam (unified batch+stream, autoscaling, windowing/watermarks).
  • Pub/Sub = global messaging (at-least-once; exactly-once option; pull/push subs).
  • GCS (storage), Dataproc (managed Spark), Composer (managed Airflow), Dataplex/Catalog (governance).
DataOps, governance & performance

🛠️ DataOps & quality

  • Observability pillars: freshness, volume, schema, distribution, lineage.
  • Quality checks: not-null, uniqueness, range/valid-set, referential — block publish on fail.
  • Beware silent failures ("succeeded" but wrong/empty data).
  • CI/CD runs tests per change; IaC (Terraform) = versioned infra; SLA/SLO = promise to consumers.

🛡️ Governance & security

  • PII = identifies a person (directly or via quasi-identifiers).
  • GDPR rights: access & erasure ("be forgotten") → needs row-level delete + lineage.
  • RBAC + least privilege; restrict at table/column/row level; masking/tokenization/pseudonymization.
  • Lineage for impact analysis; data residency = keep data in-region; encrypt at rest + in transit.

🚀 Performance & cost

  • Scan less = the #1 lever: partition, cluster/Z-order, columnar, narrow SELECT, pushdown filters.
  • Right-size files (~128–256 MB); compact small files; avoid data skew.
  • Cache/materialize hot results; prune with min/max stats; broadcast small joins.
  • Cost ∝ compute time + data scanned + egress — measure, then tune the biggest term.
DSA for data engineering

🧮 DSA — complexity & core structures

  • Big-O: array index O(1); hash map/set O(1) avg lookup/insert (dedup, joins, counting); sorted structures O(log n).
  • Hashing underlies GROUP BY, DISTINCT, hash joins & partitioning.
  • Trade time vs space (a hash map buys speed with memory).
  • Know when a scan (O(n)) beats an index for big analytical reads.

⛰️ DSA — heaps, sorting, selection

  • Heap = O(log n) push/pop → top-K (size-K heap), merge k sorted streams (heap of heads).
  • External merge sort = sort data > RAM (sort runs → k-way merge) — how warehouses sort/join at scale.
  • Quickselect = O(n) avg k-th element / median.
  • Two-heaps = streaming median; sorting enables merge joins & dedup.

🎲 DSA — probabilistic at scale

  • Bloom filter = membership in tiny memory; no false negatives, tunable false positives (dedup, skip missing keys).
  • HyperLogLog = approximate distinct count in KBs (APPROX_COUNT_DISTINCT).
  • Count-Min Sketch = approximate frequencies / heavy hitters.
  • Consistent hashing = shard/rebalance with minimal key movement when nodes change.

🌲 DSA — storage & trees

  • B-tree / B+-tree = read-optimized, balanced → RDBMS indexes, range scans.
  • LSM-tree = write-optimized: memtable → flush to SSTables + background compaction (Cassandra, RocksDB, BigTable).
  • B-tree favors reads; LSM favors high write throughput — pick by workload.
  • Trie for prefix lookups; sorted SSTables enable efficient merges & range reads.

🕸️ DSA — graphs & DAGs

  • Topological sort (Kahn's / DFS) = valid task order → Airflow/dbt/Spark DAGs.
  • Cycle detection: a pipeline DAG must be acyclic; a cycle = dependency deadlock.
  • BFS/DFS for dependency resolution, lineage traversal, reachability.
  • Model dependencies as a graph; parallelize independent branches.

🎯 DSA — coding & SQL patterns

  • Two pointers / sliding window for ordered scans & ranges.
  • Hashing for dedup, group, and O(1) joins.
  • SQL patterns: ROW_NUMBER dedup, gaps-and-islands, running totals, top-N per group, sessionization.
  • State the complexity & the data-size assumption before coding.
ML & agentic AI for DEs

🤖 ML for data engineers

  • DE owns feature pipelines + the feature store (offline train / online serve).
  • Avoid train/serve skew & label leakage (point-in-time correct features).
  • Batch vs online features; version data & features.
  • MLOps: model registry, CI/CD, monitor drift & data quality in production.

🧠 Agentic AI for DEs (2026)

  • LLMs in pipelines: extraction, classification, entity resolution, doc parsing.
  • RAG = chunk → embed → vector DB → retrieve → generate; keep embeddings fresh.
  • MCP (Model Context Protocol) connects agents to tools/data sources.
  • Evals + guardrails + cost/latency budgets make agentic systems production-safe.
System design & interview mastery

🎯 System design method

  • 1 Clarify (freshness, scale, users, history, budget).
  • 2 Estimate: vol = events×bytes; tput = events÷86,400 ×peak(2–5); storage = daily×retention×overhead.
  • 3 Flow: source → ingest → store → transform → serve.
  • 4 Choose components with reasons; 5 Hard parts: idempotency, late/dup data, quality, schema change, monitoring.
  • 6 Trade-offs: simplest design that meets the requirement (batch unless real-time is truly needed).

🌐 Distributed systems

  • Partitioning (hash/range) for scale; replication (leader-follower, quorum) for durability + reads.
  • Consistency: strong vs eventual; consensus (Raft/Paxos) for agreement.
  • Delivery: at-most / at-least / exactly-once; make sinks idempotent.
  • Fault tolerance: retries + backoff, dead-letter queues, checkpoints, backpressure.

🏗️ DE design case studies

  • Streaming analytics: Kafka → Flink/Spark SS (windowed) → serving store + dashboard.
  • CDC → lakehouse: Debezium → Kafka → MERGE into Delta/Iceberg (dedup by key + version).
  • Metrics/semantic layer: conformed dims + governed metrics for consistent KPIs.
  • Data-quality service: assertions + freshness/volume + circuit-breaker on publish.

🗣️ Interview strategy

  • The loop: SQL, coding/Python, data modeling, system design, behavioral.
  • State assumptions, think aloud, lead with the simplest correct design, then optimize.
  • Back-of-envelope the scale before choosing tools; justify every trade-off.
  • Behavioral = STAR (Situation, Task, Action, Result) with real metrics.
Rapid-fire Q&A

⚡ SQL & pipelines

Customers with no orders?

LEFT JOIN orders … WHERE order_id IS NULL.

2nd-highest salary?

DENSE_RANK() OVER(ORDER BY salary DESC), pick rank=2.

WHERE vs HAVING?

WHERE = rows pre-group; HAVING = groups post-aggregate.

Make a pipeline idempotent?

Upsert/MERGE or delete-then-insert per partition.

ETL vs ELT?

Transform before vs after loading; ELT suits cloud warehouses.

Incremental vs CDC?

High-water mark misses deletes; CDC reads the change log.

⚡ Spark, streaming & modeling

Why is shuffle slow?

Moves data across network/disk by key.

Fix Spark data skew?

Salting, AQE skew join, or broadcast the small side.

Exactly-once streaming?

Checkpoints + idempotent/transactional sink.

Star vs snowflake schema?

Denormalized dims (fewer joins) vs normalized.

SCD Type 2?

New row per change + validity dates + current flag.

Delete one user from a lake?

Table format (Delta/Iceberg) row-level DELETE + lineage.

⚡ Platforms & DSA

Why cut a BigQuery/Athena bill?

Partition + cluster + Parquet → scan fewer bytes.

Snowflake storage/compute split?

Warehouses scale compute independently of stored data.

What makes Delta ACID?

The transaction log over Parquet files.

Approx distinct count at scale?

HyperLogLog (APPROX_COUNT_DISTINCT).

Skip lookups for missing keys?

Bloom filter (no false negatives).

Order pipeline tasks safely?

Topological sort of the DAG (must be acyclic).

B-tree vs LSM-tree?

Read-optimized index vs write-optimized (SSTables + compaction).
