💡 The expression API

# everything is an expression on pl.col(...)
df.select(
    pl.col("a"),
    (pl.col("amt") - pl.col("fee")).alias("net"),
    pl.col("amt").sum().alias("total"),
)
df.with_columns(
    (pl.col("amt") * 1.1).alias("gross"),
    pl.col("name").str.to_uppercase(),
)

Expressions run in parallel and compose — the core of Polars. select = pick/derive, with_columns = add/replace.

Lazy & scale

⚙️ Lazy execution

result = (
  pl.scan_parquet("data/*.parquet")   # don't read yet
    .filter(pl.col("amt") > 0)
    .group_by("dept")
    .agg(pl.col("amt").sum())
    .collect()                       # run optimized plan
)
lf.explain()          # see the query plan
lf.collect(streaming=True)   # larger-than-RAM

Lazy = Polars pushes filters/projections into the scan (reads fewer bytes) and parallelizes. Prefer scan_* + collect().

🐼 pandas → Polars

  • No index; use columns & .filter() instead of boolean [].
  • assign/apply → with_columns(pl.col(...)) expressions.
  • groupby().agg() → group_by().agg(pl.col(...)).
  • Multi-threaded & Arrow-backed by default → often 5–30× faster, lower memory.
  • Reach for Polars/DuckDB when pandas is slow or data nears RAM limits.

⚡ Tips

  • Chain in one lazy pipeline; call collect() once at the end.
  • Select only needed columns early (projection pushdown does this for you).
  • Use native expressions over Python map_elements (slow, single-thread).
  • Read/write Parquet; it's Arrow-native and fast.
