Querying

▶ Query basics & clause order

SELECT col, other AS alias
FROM table
WHERE  pred
GROUP BY col
HAVING agg_pred
ORDER BY col DESC
LIMIT 100 OFFSET 20;

Logical run order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. So a SELECT alias can't be used in WHERE.

SELECT DISTINCT country -- unique rows
FROM customers;

-- operators
=  <>  <  <=  >  >=
AND  OR  NOT

name LIKE 'A%'   -- % = any, _ = one
name ILIKE 'a%'  -- case-insensitive (PG)
-- NULL is unknown: use IS
col IS NULL   /   col IS NOT NULL
COALESCE(col, 0)   -- first non-null

🔗 Joins

SELECT *
FROM a
JOIN       b ON a.id = b.a_id  -- INNER
LEFT JOIN  b ON a.id = b.a_id  -- +unmatched left
RIGHT JOIN b ON ...
FULL JOIN  b ON ...
CROSS JOIN b               -- cartesian
JOIN b USING (id)          -- same col name
-- anti-join: rows in a with no b
FROM a LEFT JOIN b ON a.id=b.a_id
WHERE b.a_id IS NULL;

Filter the right table in ON, not WHERE, or a LEFT JOIN silently becomes INNER. Non-unique join key multiplies rows.

Σ Aggregation & GROUP BY

SELECT dept,
       COUNT(*)          n,
       COUNT(DISTINCT uid) users,
       SUM(amt), AVG(amt),
       MIN(ts), MAX(ts)
FROM orders
GROUP BY dept
HAVING SUM(amt) > 1000;

COUNT(*) counts rows; COUNT(col) skips NULLs. Subtotals: GROUP BY ROLLUP(a,b) / CUBE / GROUPING SETS.

Analytic SQL

🪟 Window functions

func() OVER (
   PARTITION BY dept
   ORDER BY salary DESC)
ROW_NUMBER()  -- 1,2,3 unique
RANK()        -- 1,2,2,4 (gaps)
DENSE_RANK()  -- 1,2,2,3
NTILE(4)      -- quartiles
LAG(x,1) / LEAD(x,1)  -- prev/next
SUM(x) OVER(ORDER BY d)  -- running total

Top-N per group: ROW_NUMBER() in a subquery, then WHERE rn <= N. Dedup: keep rn = 1 ordered by recency.

🖼️ Window frames

AVG(x) OVER(
  ORDER BY d
  ROWS BETWEEN 6 PRECEDING
        AND CURRENT ROW)   -- 7-day avg

RANGE BETWEEN UNBOUNDED PRECEDING
      AND CURRENT ROW       -- cumulative

Default frame with ORDER BY is RANGE … UNBOUNDED PRECEDING. Use ROWS for exact row counts.

🧩 CTEs & subqueries

WITH recent AS (
  SELECT * FROM orders
  WHERE ts > CURRENT_DATE - 30
)
SELECT * FROM recent;
-- correlated + EXISTS
SELECT * FROM c
WHERE EXISTS (
  SELECT 1 FROM o WHERE o.cid=c.id);
-- recursive (hierarchies)
WITH RECURSIVE t AS (
  SELECT id,mgr FROM emp WHERE mgr IS NULL
  UNION ALL
  SELECT e.id,e.mgr FROM emp e JOIN t ON e.mgr=t.id)
SELECT * FROM t;

📅 Date & time

CURRENT_DATE   CURRENT_TIMESTAMP   NOW()

UPDATE t SET b='z' WHERE a=1;

DELETE FROM t WHERE a=1;

🔀 MERGE / upsert (idempotent)

MERGE INTO target t
USING source s ON t.id = s.id
WHEN MATCHED THEN UPDATE SET t.v = s.v
WHEN NOT MATCHED THEN INSERT (id,v)
     VALUES (s.id, s.v);
-- Postgres flavor
INSERT INTO t (id,v) VALUES (1,'a')
ON CONFLICT (id) DO UPDATE SET v = EXCLUDED.v;

MERGE / upsert = the idempotency workhorse: re-running the load yields the same result.

🔒 Transactions

BEGIN;
  UPDATE acct SET bal=bal-100 WHERE id=1;
  UPDATE acct SET bal=bal+100 WHERE id=2;
COMMIT;   -- or ROLLBACK;

ACID. Isolation levels: READ COMMITTED → REPEATABLE READ → SERIALIZABLE (fewer anomalies, more locking).

⚡ Performance

  • EXPLAIN / EXPLAIN ANALYZE — read the plan; watch seq scan vs index scan.
  • Index columns you filter/join/sort on. Don't wrap them in functions in WHERE (kills the index) — use ranges.
  • Select only needed columns; filter early; partition big tables & prune by date.
  • Prefer UNION ALL, EXISTS over IN (subquery) on big sets; watch join-key grain before SUM.
