Architecture & design

🟥 Architecture

  • Leader node plans queries; compute nodes hold data in slices and run in parallel (MPP, columnar).
  • RA3 nodes = managed storage (scale compute & storage separately); Serverless option too.
  • Performance hinges on distribution + sort keys (there are no secondary indexes).

🧭 Distribution styles

CREATE TABLE sales (
  id BIGINT, cust_id BIGINT, dt DATE, amt DECIMAL(12,2))
DISTSTYLE KEY DISTKEY(cust_id)   -- co-locate joins on cust_id
SORTKEY(dt);
  • KEY — rows with equal DISTKEY land on the same slice → fast joins/aggregations on that key.
  • ALL — full copy on every node (small dimensions).
  • EVEN — round-robin (no clear join key).
  • AUTO — Redshift decides (default).

Pick DISTKEY = the big join column; avoid skew (a hot key overloads one slice).

↕️ Sort keys & encoding

SORTKEY(dt)                -- compound (default)
INTERLEAVED SORTKEY(a, b)   -- equal weight, multi-filter

-- column compression (usually AUTO/ANALYZE picks)
amt DECIMAL(12,2) ENCODE az64,
name VARCHAR(64)  ENCODE zstd

Sort key = the column you filter on by range (usually a date) → lets Redshift skip blocks via zone maps.

Load & unload

📥 COPY from S3

COPY sales
FROM 's3://bucket/sales/'
IAM_ROLE 'arn:aws:iam::123:role/redshift'
FORMAT AS PARQUET;

-- CSV variant
COPY sales FROM 's3://bucket/f.csv'
IAM_ROLE '…' CSV IGNOREHEADER 1 GZIP
DATEFORMAT 'auto';

COPY loads in parallel across slices — far faster than INSERTs. Split files into multiples of the slice count.

📤 UNLOAD to S3

UNLOAD ('SELECT * FROM sales WHERE dt = ''2026-01-01''')
TO 's3://bucket/export/sales_'
IAM_ROLE 'arn:aws:iam::123:role/redshift'
FORMAT PARQUET
PARTITION BY (dt)
ALLOWOVERWRITE;

🔭 Spectrum (query S3)

CREATE EXTERNAL SCHEMA spectrum
FROM DATA CATALOG DATABASE 'lake'
IAM_ROLE 'arn:aws:iam::123:role/redshift';

CREATE EXTERNAL TABLE spectrum.events (
  id BIGINT, ts TIMESTAMP)
PARTITIONED BY (dt DATE)
STORED AS PARQUET LOCATION 's3://bucket/events/';
ALTER TABLE spectrum.events ADD PARTITION(dt='2026-01-01')
  LOCATION 's3://bucket/events/dt=2026-01-01/';

Join warehouse tables to open S3 data without loading it (bills per TB scanned).

Operate & tune

🧹 VACUUM & ANALYZE

VACUUM sales;             -- reclaim + re-sort after big deletes/loads
VACUUM DELETE ONLY sales;  -- just reclaim space
VACUUM SORT ONLY sales;
ANALYZE sales;            -- refresh planner stats

Deletes/updates leave dead rows & unsorted regions; VACUUM restores them. ANALYZE keeps the optimizer accurate.

🚦 Workload (WLM)

  • WLM = query queues with memory/concurrency limits; prefer Auto WLM.
  • Concurrency Scaling adds transient clusters for bursts of reads.
  • Short Query Acceleration fast-lanes small queries.
  • Set query_group / QMR rules to isolate ETL from BI.

🔍 System tables

-- recent queries
SELECT query, elapsed, substring FROM SVL_QLOG
ORDER BY starttime DESC LIMIT 20;

STL_LOAD_ERRORS       -- why a COPY failed
SVV_TABLE_INFO        -- skew, unsorted %, size
STV_WLM_QUERY_STATE   -- running queries
SVL_QUERY_REPORT      -- per-step detail
PG_TABLE_DEF          -- columns / distkey / sortkey

🔀 Upsert pattern

BEGIN;
CREATE TEMP TABLE stage (LIKE sales);
COPY stage FROM 's3://…' IAM_ROLE '…';
DELETE FROM sales USING stage WHERE sales.id = stage.id;
INSERT INTO sales SELECT * FROM stage;
COMMIT;   -- MERGE is also supported on newer clusters

⚡ Tuning checklist

  • DISTKEY on the largest join column; DISTSTYLE ALL for small dims.
  • SORTKEY on the range-filter column (date); keep tables VACUUMed.
  • Load with COPY from many gzip/Parquet files; avoid singleton INSERTs.
  • Check SVV_TABLE_INFO for skew & unsorted %; compress with ENCODE.
  • Offload cold/raw data to S3 + query via Spectrum.
