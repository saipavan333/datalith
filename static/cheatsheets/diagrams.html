🧭 Which diagram, when

Architecture .. components & how they connect
Data flow ..... how data moves & transforms
ER ............ tables, keys, relationships
Star schema ... facts & dimensions (warehouse)
Pipeline DAG .. task order & dependencies
Sequence ...... step-by-step messages over time
C4 ............ zoom levels: context->container
State ......... lifecycle of an entity/job

In a design doc or interview, lead with a simple architecture or data-flow diagram, then zoom in where it matters.

Structure & flow

🏛️ Architecture diagram

[Sources] -> [Ingest] -> [Storage] -> [Serve]
  APIs        Kafka/      Lake/        BI /
  DBs         Airflow     Warehouse    API
  • Boxes = components/services; cylinders = data stores; arrows = data/requests.
  • Label arrows (what flows, batch vs stream); group by zone/tier.
  • Goal: someone grasps the system in ~10 seconds.

🔀 Data flow diagram (DFD)

(External) --data--> [Process] --> ||Data store||
  entity              (transform)      (table/file)
  • Four elements: external entity, process (transform), data store, data flow (arrow).
  • Level 0 (context) = one process + externals; Level 1 breaks it into sub-processes.
  • Focus on what happens to the data, not the tech.

🗝️ ER diagram (crow's foot)

CUSTOMER ||--o{ ORDER : places
ORDER    ||--|{ ORDER_ITEM : contains

|| one (and only one)   o{ zero or many
|{ one or many          o| zero or one
  • Entities = tables; lines = relationships; the crow's foot (fork) = the "many" side.
  • Mark PK and FK; a many-to-many resolves into a junction table.
  • Used for OLTP schema design & documenting sources.

⭐ Star / snowflake schema

   dim_date   dim_customer
        \        /
        [ FACT_SALES ]      (measures + FKs)
        /        \
  dim_product   dim_store
  • Fact in the center (events/measures + dimension keys); dimensions around it.
  • Star = denormalized dims; snowflake = dims normalized into sub-tables.
  • The go-to diagram for warehouse/dimensional design.

🗺️ Pipeline DAG

extract -> validate -> transform -> load -> publish
                 \-> quarantine (on fail)
  • Nodes = tasks; directed edges = dependencies; must be acyclic.
  • Shows order, parallel branches, and ret/failure paths (Airflow/dbt/Dagster).
  • Independent branches run in parallel; a cycle = a design bug.
Behavior & zoom

⏱️ Sequence diagram

Client -> API:    request
API    -> Kafka:  produce event
Kafka  -> Worker: consume
Worker -> DB:     upsert
API    --> Client: 202 Accepted   (dashed = response)
  • Vertical lifelines per actor; time flows downward; arrows = messages.
  • Solid = call/sync, dashed = return/async; great for API/event flows & race conditions.

🔎 C4 model (zoom levels)

  • 1 Context — the system as one box + users & external systems.
  • 2 Container — apps, services, databases inside the system.
  • 3 Component — the parts inside one container.
  • 4 Code — classes (rarely drawn).

Start broad (context), zoom only where detail adds value — perfect structure for a design review.

🔁 State diagram

[*] -> queued -> running -> success -> [*]
                    running -> failed -> retrying

Models the lifecycle of an entity or job (states + transitions) — handy for order status, job runs, CDC records.

🎯 In a system-design interview

  • Draw the data-flow / architecture first: source → ingest → store → transform → serve.
  • Mark data stores (queue, lake, warehouse) and label batch vs stream on arrows.
  • Call out the hard parts on the diagram: idempotency, late/dup data, partitioning, schema change.
  • Zoom into one component (C4-style) when asked; keep the top-level picture legible.

💡 Tips

  • Label every arrow (what data, direction, batch/stream); add a small legend.
  • Left→right or top→down flow; consistent shapes (box=service, cylinder=store, queue=pipe).
  • One level of detail per diagram — split rather than cram.
  • Keep it in version control as code with Mermaid (see that cheat sheet).
