🔺 Delta & Iceberg Cheat Sheet

Open lakehouse table formats — the transaction log that gives files ACID, MERGE, time travel and schema evolution. Delta vs Iceberg vs Hudi.

← All cheat sheets Back to app
Concept

📖 What & why

  • A table format = Parquet data files + a transaction log / metadata layer over them.
  • Gives a plain file lake: ACID writes, row-level UPDATE/DELETE/MERGE (GDPR), time travel, schema evolution, concurrent readers/writers.
  • Delta → _delta_log (JSON + checkpoints). Iceberg → metadata/manifest tree. Hudi → timeline + copy-on-write / merge-on-read.
Delta Lake

🔺 Delta — create & write

-- SQL
CREATE TABLE t (id BIGINT, amt DOUBLE, dt DATE)
USING DELTA PARTITIONED BY (dt);

-- PySpark
df.write.format("delta").mode("append").saveAsTable("t")
df.write.format("delta").mode("overwrite") \
  .option("replaceWhere", "dt = '2026-01-01'").save(path)

🔀 Delta — MERGE & delete

MERGE INTO t USING updates u ON t.id = u.id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

DELETE FROM t WHERE dt < '2025-01-01';   -- row-level
UPDATE t SET amt = 0 WHERE amt IS NULL;

🧹 Delta — maintain & travel

OPTIMIZE t ZORDER BY (id);   -- compact + co-locate
VACUUM t RETAIN 168 HOURS;   -- purge old files

SELECT * FROM t VERSION AS OF 10;
SELECT * FROM t TIMESTAMP AS OF '2026-01-01';
DESCRIBE HISTORY t;
RESTORE TABLE t TO VERSION AS OF 10;
Apache Iceberg

🧊 Iceberg — create & write

CREATE TABLE db.t (id bigint, amt double, ts timestamp)
USING iceberg
PARTITIONED BY (days(ts));   -- hidden partitioning

INSERT INTO db.t SELECT * FROM staging;
df.writeTo("db.t").append()        # Spark 3
df.writeTo("db.t").overwritePartitions()

Hidden partitioning: query WHERE ts > … and Iceberg prunes partitions automatically — no separate partition column to filter.

🕰️ Iceberg — travel & evolve

-- time travel
SELECT * FROM db.t FOR VERSION AS OF 42;
SELECT * FROM db.t FOR TIMESTAMP AS OF '2026-01-01';

-- schema & partition evolution (no rewrite)
ALTER TABLE db.t ADD COLUMN region string;
ALTER TABLE db.t ADD PARTITION FIELD bucket(16, id);

-- inspect metadata tables
SELECT * FROM db.t.snapshots;
SELECT * FROM db.t.history;

🧰 Iceberg — maintenance

MERGE INTO db.t ... ;   -- ACID upsert (like Delta)

-- housekeeping procedures
CALL cat.system.expire_snapshots('db.t');
CALL cat.system.rewrite_data_files('db.t');   -- compaction
CALL cat.system.rewrite_manifests('db.t');

Engine-agnostic: Spark, Flink, Trino, Snowflake & others can read/write the same Iceberg table.

Choosing

⚖️ Delta vs Iceberg vs Hudi

Delta    tight w/ Spark & Databricks; simple;
         _delta_log; OPTIMIZE/ZORDER; CDF.
Iceberg  engine-neutral (Trino/Flink/Snowflake);
         hidden partitioning; partition & schema
         evolution; snapshot metadata tables.
Hudi     record-level upserts/CDC; copy-on-write
         (read-fast) vs merge-on-read (write-fast).

All three give ACID + time travel + upserts. Pick by engine ecosystem and who else must read the data.

⚠️ Gotchas

  • Run compaction (OPTIMIZE / rewrite_data_files) — streaming writes make tiny files.
  • VACUUM / expire_snapshots reclaim storage but shorten time-travel — keep enough retention.
  • Concurrent writers need the format's commit protocol (a metastore/catalog) — don't hand-edit files.
  • Partition on low-cardinality columns (date), not high-cardinality ids.
