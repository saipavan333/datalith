# 06 · Analytics engineering with dbt

A real **dbt** project on **DuckDB**: a staging view + a mart table wired by `ref()`, with **data tests** — the modern
ELT / analytics-engineering workflow.

```bash
pip install -r requirements.txt   # dbt-duckdb
python run.py                      # generates a seed, then dbt seed -> run -> test
# or run dbt directly:
DBT_PROFILES_DIR=. dbt build
```

## Project layout

```
models/staging/stg_orders.sql      -- clean/conform (view)        ref('raw_orders')
models/marts/orders_by_region.sql  -- business aggregate (table)  ref('stg_orders')
models/schema.yml                  -- not_null / unique / accepted_values tests
seeds/raw_orders.csv               -- generated by run.py
profiles.yml                       -- DuckDB target (out/warehouse.duckdb)
```

## What it shows

- **`ref()` builds the DAG** — dbt knows `orders_by_region` depends on `stg_orders` depends on `raw_orders`.
- **Materializations** — staging as a view, mart as a table.
- **Tests** — `not_null`, `unique`, `accepted_values` enforce quality (the staging layer also drops invalid rows).
- **Docs/lineage** — `dbt docs generate && dbt docs serve` renders the lineage graph.

## Production mapping

Swap the DuckDB profile for Snowflake/BigQuery/Redshift (`dbt-snowflake`, etc.); add sources, snapshots (SCD2),
incremental models, and run `dbt build` in CI.
